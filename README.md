## ส่วนที่ 0: ทำความเข้าใจก่อนเริ่ม

อย่าเพิ่งพิมพ์คำสั่งอะไรในส่วนนี้ — อ่านทำความเข้าใจก่อน เพราะถ้าเข้าใจภาพรวมตรงนี้ ขั้นตอนที่เหลือจะง่ายขึ้นเยอะ

### Git คืออะไร (พูดง่ายๆ)

ลองนึกภาพว่าเรากำลังเขียนรายงานใน Word แล้วเซฟไฟล์ทับของเดิมไปเรื่อยๆ — พอวันหนึ่งอยากย้อนกลับไปดูเวอร์ชัน
เมื่อวาน ก็ทำไม่ได้แล้วเพราะเซฟทับไปแล้ว

**Git** คือโปรแกรมที่แก้ปัญหานี้ มันจะ "ถ่ายรูป" โค้ดทั้งหมด ณ ช่วงเวลาที่เราสั่งให้มันถ่าย (เรียกว่า **commit**)
พร้อมจดโน้ตกำกับไว้ว่า "รูปนี้ถ่ายตอนไหน แก้อะไรไป" ทำให้ย้อนดู หรือย้อนกลับไปเวอร์ชันเก่าได้เสมอ

Git เป็นโปรแกรมที่ **ติดตั้งอยู่ในเครื่องคุณ** ทำงานได้แม้ไม่ต่อเน็ต

### GitHub คืออะไร

**GitHub** คือเว็บไซต์ (github.com) ที่เก็บ "อัลบั้มรูปที่ Git ถ่ายไว้" ให้อยู่บนอินเทอร์เน็ต

ลองเทียบง่ายๆ:
- Git เหมือนกล้องถ่ายรูป + อัลบั้มรูปในเครื่องคุณ
- GitHub เหมือน Google Photos ที่เอาอัลบั้มนั้นไปเก็บบนคลาวด์ ให้คนอื่นเปิดดูได้ด้วย

การ "**push**" คือการอัปโหลดรูปจากอัลบั้มในเครื่อง ขึ้นไปเก็บที่ GitHub

### GitHub Pages คืออะไร

**GitHub Pages** คือฟีเจอร์ฟรีของ GitHub ที่หยิบไฟล์ HTML/CSS/JavaScript ที่เก็บไว้ มาทำเป็น
**เว็บไซต์จริงที่มีลิงก์เปิดได้** โดยที่เราไม่ต้องไปเช่า web server เอง ไม่ต้องรู้เรื่อง server เลยด้วยซ้ำ

สรุปห่วงโซ่ทั้งหมด: **แก้ไฟล์ในเครื่อง → Git บันทึกเป็นเวอร์ชัน (commit) → Push ขึ้น GitHub → GitHub Pages
แปลงเป็นเว็บไซต์ให้อัตโนมัติ**

### ทำไมต้องทำขนาดนี้ เปิดไฟล์ตรงๆ ไม่ได้เหรอ

เปิดไฟล์ตรงในเครื่องได้ แต่ **เปิดได้แค่คนเดียวคือเจ้าของเครื่อง** เพื่อนร่วมทีมคนอื่นเข้าไม่ถึง
การทำตามคู่มือนี้จะได้ **ลิงก์เว็บไซต์จริง** ที่ใครก็เปิดได้จากที่ไหนก็ได้ ผ่านเบราว์เซอร์ธรรมดา
ไม่มีข้อจำกัดเรื่องดาวน์โหลดไฟล์แบบที่เจอตอนใช้ Claude Artifact preview ด้วย

### สิ่งที่ต้องมีก่อนเริ่ม

- Mac (คู่มือนี้ใช้คำสั่งสำหรับ macOS)
- อีเมลไว้สมัครบัญชี GitHub (ถ้ายังไม่มีบัญชี)
- ไฟล์โปรเจกต์ที่อยากเอาขึ้นเว็บ (ในตัวอย่างนี้คือโฟลเดอร์ `/Users/natingkaninantana/Automation`
  ที่มีไฟล์ `crossformatstep.html`)

## ส่วนที่ 1: เปิด Terminal ให้เป็น

**Terminal** คือแอปที่ใช้พิมพ์คำสั่งสั่งงานคอมพิวเตอร์โดยตรง (แทนการคลิกเมาส์) เราจะใช้มันพิมพ์คำสั่ง Git ทั้งหมด

### วิธีเปิด

1. กด **Cmd + Space** (ปุ่ม Command ค้างไว้ แล้วกด Spacebar) — จะมีช่องค้นหา (Spotlight) เด้งขึ้นมากลางจอ
2. พิมพ์คำว่า `Terminal`
3. กด **Enter**
4. จะได้หน้าต่างสีดำ/ขาว มีข้อความรอพิมพ์อยู่ แบบนี้:

   ```
   yourname@Mac ~ %
   ```

   ส่วนที่ขึ้นก่อนเครื่องหมาย `%` (หรือ `$` ในบางเครื่อง) คือ "ตำแหน่งปัจจุบัน" ที่เราอยู่ ไม่ต้องตกใจถ้าหน้าตาไม่เหมือนกันเป๊ะ

### วิธีใช้งานเบื้องต้นที่ต้องรู้

- **พิมพ์คำสั่งแล้วกด Enter** คำสั่งถึงจะทำงาน (พิมพ์เฉยๆ ไม่กด Enter จะยังไม่เกิดอะไรขึ้น)
- **Copy/Paste ใช้ได้ปกติ** — Cmd+C คัดลอก, Cmd+V วาง เหมือนแอปอื่นๆ
- **`cd` แปลว่า "เข้าไปในโฟลเดอร์"** (ย่อจาก change directory) เช่น

  ```bash
  cd /Users/natingkaninantana/Automation
  ```

  แปลว่า "ย้ายเข้าไปทำงานในโฟลเดอร์ Automation" คำสั่งต่อจากนี้ทั้งหมดจะมีผลกับไฟล์ในโฟลเดอร์นี้เท่านั้น

- ถ้าพิมพ์ผิด กด Enter ไปแล้ว มักจะมีข้อความสีแดงหรือคำว่า `error` / `not found` ขึ้นมา — ไม่ใช่เรื่องแปลก
  อ่านข้อความดูว่ามันบอกว่าอะไร (ในคู่มือนี้จะพาไปดูข้อความ error จริงที่อาจเจอ ในส่วนที่ 10)

**ทุกคำสั่งในคู่มือนี้ ให้เปิด Terminal ค้างไว้หน้าต่างเดิม แล้วพิมพ์ต่อกันไปได้เลยตามลำดับ**

## ส่วนที่ 3: สร้างกุญแจดิจิทัล (SSH Key)

### ทำไมต้องมีขั้นตอนนี้

GitHub **ไม่อนุญาต**ให้ส่งไฟล์ขึ้น (push) ด้วยการพิมพ์ username/password ธรรมดาอีกต่อไปแล้ว
(ยกเลิกไปหลายปีแล้วด้วยเหตุผลด้านความปลอดภัย) ต้องใช้ **SSH key** แทน

ลองเทียบ SSH key เหมือน **กุญแจ 2 ดอกที่เข้าคู่กัน**:
- **Private key** (กุญแจไข) — อยู่ในเครื่องคุณเท่านั้น **ห้ามให้ใครดู ห้ามส่งให้ใครเด็ดขาด**
- **Public key** (กุญแจล็อก/แม่กุญแจ) — เอาไปให้ GitHub เก็บไว้ได้อย่างปลอดภัย ใครเห็นก็ไม่เป็นไร

พอทั้งสองฝั่งมีกุญแจที่เข้าคู่กัน เวลา push GitHub จะรู้ทันทีว่า "นี่คือเครื่องของเจ้าของบัญชีจริง" โดยไม่ต้องพิมพ์
password ทุกครั้ง

### ขั้นตอนสร้างกุญแจ

พิมพ์คำสั่งนี้ใน Terminal (คัดลอกไปวางได้เลยทั้งบรรทัด):

```bash
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -N "" -C "ชื่ออะไรก็ได้ให้จำง่าย เช่น my-mac"
```

**คำสั่งนี้แปลว่าอะไรทีละส่วน:**
- `ssh-keygen` = โปรแกรมสำหรับสร้างกุญแจ SSH
- `-t ed25519` = ใช้เทคนิคการเข้ารหัสแบบ ed25519 (มาตรฐานที่แนะนำในปัจจุบัน)
- `-f ~/.ssh/id_ed25519` = บอกว่าให้เก็บกุญแจไว้ที่ไหน (`~/.ssh/` คือโฟลเดอร์ซ่อนสำหรับเก็บกุญแจโดยเฉพาะ)
- `-N ""` = ไม่ต้องตั้งรหัสผ่านเพิ่มสำหรับกุญแจนี้ (เพื่อความง่าย)
- `-C "..."` = ใส่ label/หมายเหตุกำกับกุญแจ จะได้รู้ภายหลังว่ากุญแจนี้มาจากเครื่องไหน

กด Enter แล้วรอ 1-2 วินาที จะเห็นข้อความประมาณนี้ (ตัวอักษร/ตัวเลขจะไม่เหมือนกันทุกครั้ง ไม่เป็นไร):

```
Your identification has been saved in /Users/yourname/.ssh/id_ed25519
Your public key has been saved in /Users/yourname/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx my-mac
```

แปลว่าสร้างกุญแจสำเร็จแล้ว ตอนนี้มีไฟล์ 2 ไฟล์ในเครื่อง:
- `id_ed25519` = กุญแจไข (private) — **ห้ามเปิดให้ใครดู**
- `id_ed25519.pub` = กุญแจล็อก (public) — เอาไปให้ GitHub ได้

### เอา Public Key ไปวางที่ GitHub

1. พิมพ์คำสั่งนี้เพื่อดูเนื้อหากุญแจล็อก:

   ```bash
   cat ~/.ssh/id_ed25519.pub
   ```

2. จะได้ข้อความยาวๆ หนึ่งบรรทัด ขึ้นต้นด้วย `ssh-ed25519 AAAA...` **คัดลอกทั้งบรรทัดนี้ไว้** (ลากเมาส์เลือก
   แล้ว Cmd+C)

3. เปิดเบราว์เซอร์ ไปที่ **[github.com/settings/ssh/new](https://github.com/settings/ssh/new)**

4. หน้าเว็บจะมีช่องให้กรอก:
   - **Title**: ตั้งชื่ออะไรก็ได้ให้จำได้ เช่น "MacBook ของฉัน"
   - **Key type**: ปล่อยเป็นค่า default (Authentication Key)
   - **Key**: วางข้อความที่คัดลอกมา (Cmd+V)

5. กดปุ่มสีเขียว **Add SSH key**

### ทดสอบว่าเชื่อมต่อสำเร็จ

กลับมาที่ Terminal พิมพ์:

```bash
ssh -T git@github.com
```

ครั้งแรกอาจมีข้อความถามว่า "Are you sure you want to continue connecting?" — พิมพ์ `yes` แล้ว Enter

ถ้าสำเร็จ จะเห็นข้อความแบบนี้:

```
Hi <username>! You've successfully authenticated, but GitHub does not provide shell access.
```

ข้อความนี้ **ไม่ใช่ error** — คำว่า "does not provide shell access" เป็นเรื่องปกติ แปลว่าเชื่อมต่อสำเร็จแล้ว
(ถ้าเจอปัญหาตรงนี้ ดูวิธีแก้ในส่วนที่ 10)

## ส่วนที่ 4: สร้าง Repository บนเว็บ GitHub

**Repository** (เรียกสั้นๆ ว่า "repo") คือ "กล่องเก็บโปรเจกต์" หนึ่งกล่องบน GitHub — 1 โปรเจกต์ = 1 repo

1. ไปที่ **[github.com/new](https://github.com/new)** (ต้อง login ค้างไว้ก่อน)

2. หน้าเว็บจะมีฟอร์มให้กรอก:
   - **Repository name**: ตั้งชื่อโปรเจกต์ เช่น `crossformat-tool` (ใช้ตัวเล็ก ขีดกลาง ไม่มีเว้นวรรค)
   - **Description** (ไม่บังคับ): ใส่คำอธิบายสั้นๆ ก็ได้ ไม่ใส่ก็ได้
   - เลือก **Public** — วงกลมด้านบน (สำคัญ! ต้องเป็น Public ถึงจะใช้ GitHub Pages ได้ฟรี
     ถ้าเลือก Private ต้องเสียเงินซื้อแพ็กเกจ GitHub Pro/Team ถึงจะเปิด Pages ได้)
   - หัวข้อ "Initialize this repository with:" — **ปล่อยว่างทุกช่อง อย่าติ๊ก**
     (ไม่ติ๊ก Add a README file, ไม่ติ๊ก .gitignore, ไม่เลือก license) เพราะถ้าติ๊กไว้ จะทำให้ตอน push
     จากเครื่องมีปัญหาไฟล์ชนกัน

3. กดปุ่มสีเขียว **Create repository** ด้านล่าง

4. จะพาไปหน้า repo ที่ยังว่างเปล่า มีกล่องคำสั่งแนะนำ — **ไม่ต้องทำตามในหน้านั้น** เดี๋ยวคู่มือนี้จะบอกคำสั่งเองในส่วนถัดไป

5. **สิ่งที่ต้องจำจากหน้านี้**: URL ของ repo รูปแบบ

   ```
   git@github.com:ชื่อบัญชีคุณ/crossformat-tool.git
   ```

   (ดูได้จากปุ่ม **Code** สีเขียวที่หน้า repo แล้วเลือกแท็บ **SSH** จะเห็น URL นี้ พร้อมปุ่ม copy)

## ส่วนที่ 5: เปลี่ยนโฟลเดอร์ในเครื่องให้เป็น Git Repo

กลับมาที่ Terminal เข้าไปที่โฟลเดอร์โปรเจกต์ก่อน (เปลี่ยน path ตามที่คุณเก็บไฟล์จริง):

```bash
cd /Users/natingkaninantana/Automation
```

### สั่งให้ Git เริ่มติดตามโฟลเดอร์นี้

```bash
git init
```

จะเห็นข้อความ:

```
Initialized empty Git repository in /Users/natingkaninantana/Automation/.git/
```

แปลว่า Git เริ่ม "จับตาดู" โฟลเดอร์นี้แล้ว (มันสร้างโฟลเดอร์ซ่อนชื่อ `.git` ไว้เก็บประวัติทั้งหมด
ไม่ต้องไปยุ่งกับโฟลเดอร์นี้เอง)

### บอก Git ว่าไฟล์ไหนไม่ต้องสนใจ

บางไฟล์ในเครื่องเป็นไฟล์ระบบ/ไฟล์ส่วนตัว ไม่ควรเอาขึ้น GitHub (เช่นไฟล์ที่ macOS สร้างเองอัตโนมัติ)
เราจะสร้างไฟล์พิเศษชื่อ `.gitignore` เพื่อบอก Git ว่า "ไฟล์พวกนี้ไม่ต้องแตะ"

พิมพ์คำสั่งนี้เพื่อสร้างไฟล์นั้นเลย:

```bash
cat > .gitignore << 'EOF'
.DS_Store
.claude/
EOF
```

(ไม่ต้องเข้าใจ syntax นี้ทั้งหมดก็ได้ แค่รู้ว่ามันสร้างไฟล์ `.gitignore` ที่มีเนื้อหา 2 บรรทัดนั้นให้)

## ส่วนที่ 6: บันทึกไฟล์เป็นเวอร์ชัน (Commit)

การ commit มี 2 ขั้นตอนย่อยเสมอ: **add** (เลือกไฟล์) แล้ว **commit** (บันทึกจริง)

### ขั้นตอนที่ 1: เลือกไฟล์ที่จะบันทึก (add)

ก่อนอื่นเช็คก่อนว่ามีไฟล์อะไรเปลี่ยนแปลงบ้าง:

```bash
git status
```

จะเห็นรายชื่อไฟล์เป็นสีแดง ภายใต้หัวข้อ "Untracked files" — คือไฟล์ที่ Git เห็นแล้วแต่ยังไม่ได้ถูกสั่งให้บันทึก

เลือกไฟล์ที่ต้องการ (ระบุชื่อไฟล์ตรงๆ ปลอดภัยกว่าการเลือกทั้งหมด):

```bash
git add crossformatstep.html index.html CLAUDE.md .gitignore
```

รัน `git status` อีกครั้งจะเห็นชื่อไฟล์เปลี่ยนเป็นสีเขียว ภายใต้หัวข้อ "Changes to be committed" —
แปลว่าเตรียมพร้อมจะบันทึกแล้ว (แต่ยังไม่ได้บันทึกจริง)

### ขั้นตอนที่ 2: บันทึกจริง (commit)

```bash
git commit -m "First commit: เพิ่มเครื่องมือ CrossFormat"
```

- `-m "..."` คือข้อความอธิบายว่า commit นี้ทำอะไร (**m** ย่อจาก message) — เขียนสั้นๆ ให้เข้าใจว่าทำอะไรไป
  เพราะภายหลังเวลาย้อนดูประวัติ จะได้รู้ว่าแต่ละจุดแก้อะไรบ้าง

จะเห็นข้อความสรุปแบบนี้:

```
[main (root-commit) xxxxxxx] First commit: เพิ่มเครื่องมือ CrossFormat
 4 files changed, 5368 insertions(+)
```

แปลว่าบันทึกสำเร็จแล้ว 1 เวอร์ชัน (เรียกว่า 1 commit) — แต่ตอนนี้ยังอยู่แค่ในเครื่องคุณเท่านั้น
ยังไม่ได้ขึ้น GitHub

## ส่วนที่ 7: ผูกกับ GitHub และ Push

### ผูก repo ในเครื่องเข้ากับ repo บนเว็บ

```bash
git remote add origin git@github.com:ชื่อบัญชีคุณ/crossformat-tool.git
```

(แทน `ชื่อบัญชีคุณ` ด้วย username จริง และ URL ต้องตรงกับที่คัดลอกมาจากส่วนที่ 4)

- `remote` แปลว่า "ปลายทางระยะไกล" (คือ repo บน GitHub)
- `origin` คือชื่อเรียกปลายทางนั้น (ตั้งชื่ออะไรก็ได้ แต่ทุกคนใช้ชื่อนี้เป็นธรรมเนียม)

คำสั่งนี้จะไม่มีข้อความอะไรขึ้นมาถ้าสำเร็จ (เงียบ = ปกติ)

### Push — ส่งขึ้น GitHub จริงๆ

```bash
git push -u origin main
```

- `push` = ส่งขึ้น
- `origin` = ไปที่ปลายทางชื่อนี้ (ที่ผูกไว้ขั้นตอนก่อนหน้า)
- `main` = ชื่อ branch (เส้นสายของโค้ด — โปรเจกต์นี้ใช้เส้นเดียวชื่อ main)
- `-u` = ใช้แค่ครั้งแรก เพื่อ "จำคู่" ไว้ว่า branch main ในเครื่อง ผูกกับ branch main บน GitHub
  ตัวไหน ครั้งต่อไปจะได้พิมพ์แค่ `git push` เฉยๆ ไม่ต้องพิมพ์ยาว

จะเห็นข้อความประมาณนี้ตอน push สำเร็จ:

```
To github.com:username/crossformat-tool.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

**ลองรีเฟรชหน้า repo บนเว็บ GitHub ดู** — จะเห็นไฟล์ทั้งหมดขึ้นไปอยู่บนนั้นแล้ว

## ส่วนที่ 8: เปิดใช้งาน GitHub Pages

ตอนนี้ไฟล์อยู่บน GitHub แล้ว แต่ยังเป็นแค่ "ที่เก็บไฟล์" ยังไม่ใช่ "เว็บไซต์" ต้องเปิดสวิตช์อีกนิดเดียว

1. ไปที่หน้า repo บนเว็บ GitHub (`github.com/ชื่อบัญชีคุณ/crossformat-tool`)

2. คลิกแท็บ **Settings** (อยู่แถบเมนูด้านบนของหน้า repo ขวาสุด มีไอคอนรูปเฟือง ⚙️)

3. ในเมนูด้านซ้ายของหน้า Settings เลื่อนหาแล้วคลิก **Pages**

4. หัวข้อ **"Build and deployment"** → ที่ dropdown **Source** เลือก **Deploy from a branch**

5. ด้านล่างจะมี dropdown ให้เลือก **Branch** — เลือก **main** และช่องข้างๆ เลือก **/ (root)**

6. กดปุ่ม **Save**

7. รอประมาณ **1-2 นาที** (มีแถบข้อความบอกสถานะ "Your site is live at...")

8. รีเฟรชหน้าเดิม จะเห็นลิงก์ขึ้นมาด้านบนแบบนี้:

   ```
   https://ชื่อบัญชีคุณ.github.io/crossformat-tool/
   ```

**เปิดลิงก์นี้ในเบราว์เซอร์ดู** — ถ้าเห็นหน้าเครื่องมือขึ้นมาปกติ แปลว่าสำเร็จทั้งหมดแล้ว 🎉
ลิงก์นี้ส่งให้ทีมเปิดใช้งานได้เลย ไม่ต้องติดตั้งอะไรเพิ่ม

## ส่วนที่ 10: เวลาอะไรพัง ทำยังไง (Troubleshooting)

### `Host key verification failed`

**แปลว่า:** เครื่องนี้ยังไม่เคย "รู้จัก" github.com มาก่อน (เป็นการเชื่อมต่อ SSH ครั้งแรก)

**วิธีแก้:**
```bash
ssh-keyscan github.com >> ~/.ssh/known_hosts
```
แล้วลองคำสั่งเดิมอีกครั้ง

### `Permission denied (publickey)`

**แปลว่า:** ยังไม่ได้เอา public key ไปวางในบัญชี GitHub (หรือวางผิดกุญแจ)

**วิธีแก้:** กลับไปทำ [ส่วนที่ 3](#ส่วนที่-3-สร้างกุญแจดิจิทัล-ssh-key) หัวข้อ "เอา Public Key ไปวางที่ GitHub"
ให้ครบ แล้วเช็คด้วย `ssh -T git@github.com` อีกครั้ง

### `fatal: not a git repository`

**แปลว่า:** ยังไม่ได้รัน `git init` ในโฟลเดอร์นี้ หรืออยู่ผิดโฟลเดอร์

**วิธีแก้:** เช็คว่าอยู่โฟลเดอร์ถูกไหมด้วยคำสั่ง `pwd` (แสดง path ปัจจุบัน) แล้วรัน `cd` ไปโฟลเดอร์ที่ถูกต้อง

### `git push` ขึ้นบอกว่า "failed to push some refs" / "rejected"

**แปลว่า:** บน GitHub มีอะไรบางอย่างที่เครื่องเรายังไม่มี (เช่น ตอนสร้าง repo ไปติ๊ก "Add a README" ไว้)

**วิธีแก้:** ถ้ามั่นใจว่าไฟล์ในเครื่องคือเวอร์ชันล่าสุดที่ถูกต้อง ใช้คำสั่ง (ใช้ด้วยความระมัดระวัง เพราะจะเขียนทับ
ประวัติบน GitHub):
```bash
git push -u origin main --force
```
ถ้าไม่แน่ใจ ให้หยุดและสอบถามก่อน อย่าเพิ่งใช้ `--force` เอง

### เว็บ GitHub Pages ขึ้น 404 หรือยังไม่อัปเดต

**สาเหตุที่พบบ่อย:** รอไม่ถึง 1-2 นาทีหลัง push หรือหลังกด Save ที่หน้า Pages

**วิธีแก้:** รอสัก 2-3 นาทีแล้วลอง Hard Refresh หน้าเว็บ (Cmd+Shift+R บน Mac) เพื่อเลี่ยง cache เก่า

## ส่วนที่ 11: ใบโกงคำสั่งทั้งหมด

รวมทุกคำสั่งไว้ในที่เดียว สำหรับก็อปไปใช้เร็วๆ (ตั้งค่าครั้งแรกให้ทำตามลำดับบนลงล่าง)

```bash
# ครั้งแรกเท่านั้น: สร้างกุญแจ SSH
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -N "" -C "my-mac"
cat ~/.ssh/id_ed25519.pub
# --> เอาผลลัพธ์ไปวางที่ github.com/settings/ssh/new

# ครั้งแรกเท่านั้น: ทดสอบการเชื่อมต่อ
ssh -T git@github.com

# ครั้งแรกเท่านั้น: ตั้งค่า repo ในเครื่อง
cd /Users/natingkaninantana/Automation
git init
cat > .gitignore << 'EOF'
.DS_Store
.claude/
EOF
git add crossformatstep.html index.html CLAUDE.md .gitignore
git commit -m "First commit"
git remote add origin git@github.com:ชื่อบัญชีคุณ/crossformat-tool.git
git push -u origin main

# ทุกครั้งที่แก้โค้ดในอนาคต — ทำแค่นี้พอ
cd /Users/natingkaninantana/Automation
git add crossformatstep.html
git commit -m "อธิบายว่าแก้อะไร"
git push
```

## ศัพท์ที่ควรรู้ (Glossary)

| คำ | ความหมาย |
|---|---|
| **Terminal** | แอปสำหรับพิมพ์คำสั่งสั่งงานคอมพิวเตอร์ |
| **repository (repo)** | "กล่องเก็บโปรเจกต์" หนึ่งกล่อง พร้อมประวัติการแก้ไขทั้งหมด |
| **commit** | การบันทึก "จุดเวลา" หนึ่งของโค้ด พร้อมข้อความอธิบายว่าทำไมถึงแก้ |
| **push** | ส่งการเปลี่ยนแปลงจากเครื่องเรา ขึ้นไปเก็บที่ GitHub |
| **pull** | ดึงการเปลี่ยนแปลงจาก GitHub ลงมาที่เครื่อง (เผื่อมีคนอื่นแก้ไว้) |
| **branch** | เส้นแยกของโค้ด (โปรเจกต์นี้ใช้แค่เส้นเดียวชื่อ `main`) |
| **remote** | ชื่อเรียกของ repo ปลายทางบน GitHub (ตั้งชื่อว่า `origin`) |
| **SSH key** | กุญแจดิจิทัลไว้ยืนยันตัวตนกับ GitHub แทนการพิมพ์ password |
| **GitHub Pages** | ฟีเจอร์ฟรีที่แปลงไฟล์ในเว็บ GitHub ให้เป็นเว็บไซต์ที่มีลิงก์เปิดได้จริง |
